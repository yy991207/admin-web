# 改动背景

这轮主要是排查当前管理后台里已经存在的潜在 bug，并做最小范围修复。先用 `lint` 和 `build` 做静态排查，再结合页面和 service 实现做了一轮人工检查，优先处理会导致编译失败、接口请求异常和页面编辑异常的问题。

# 主要修改点

## 1. 修复前端资源类型声明缺失导致的编译失败

- 新增了 `src/vite-env.d.ts`
- 补上了 `vite/client`、`*.yaml?raw` 的类型声明
- 解决了 `less`、`css`、`yaml?raw` 相关资源在 TypeScript 构建阶段无法识别的问题

## 2. 统一管理后台请求封装，补上鉴权头

- 修改了 `src/services/config.ts`
- 新增了统一的 `ApiResponse<T>` 和 `request<T>()`
- 在统一请求里补上 `Authorization: Bearer <token>`
- 关键原因是项目说明里已经约定所有 service 都应该走 `config.ts` 的统一请求封装，但当前多个 service 各自实现了 `request()`，实际没有带 token，联调时很容易直接报 401

涉及模块：

- `src/services/agentService.ts`
- `src/services/commandService.ts`
- `src/services/practiceService.ts`
- `src/services/skillService.ts`
- `src/services/templateService.ts`

同时补了两个直接 `fetch` 的接口鉴权头：

- `src/services/skillService.ts` 里的 zip 导入
- `src/services/documentParseService.ts` 里的文档解析提交和状态查询

## 3. 修复文件树编辑里的潜在问题

- 修改了 `src/pages/SystemSkills/SystemSkills.tsx`

具体修复点：

- 把不安全的 `value.hasOwnProperty(...)` 改成 `Object.prototype.hasOwnProperty.call(...)`
- 重命名时禁止输入 `/`，避免把单个文件名误改成非法路径
- 文件夹重命名时增加重名冲突检查，避免覆盖已有路径
- 文件夹重命名后同步更新当前选中的文件路径，避免右侧编辑器指向旧路径

这部分属于真实运行时风险，不只是 lint 问题。原来如果重命名文件夹，右侧编辑区可能还指向旧路径，继续编辑时会出现状态不一致。

## 4. 修复文档解析轮询里的类型和状态问题

- 修改了 `src/services/documentParseService.ts`

具体修复点：

- 轮询进度回调里把 `null` 进度转成 `undefined`
- 避免把 `null` 直接传给只接受 `number | undefined` 的回调，减少进度状态异常

## 5. 清理无效代码，消除 lint 阻塞

- 修改了以下页面中的未使用导入、未使用方法和 Hook 依赖问题：
  - `src/pages/Agents/Agents.tsx`
  - `src/pages/ClawHub/ClawHub.tsx`
  - `src/pages/Practices/Practices.tsx`
  - `src/pages/SystemSkills/SystemSkills.tsx`

## 6. 修复上传模块在生产构建里的配置读取问题

- 修改了 `src/services/ossUploadService.ts`

具体修复点：

- 删除了运行时 `fetch('/config.yaml')` 的逻辑
- 直接复用 `src/services/config.ts` 里已经解析好的配置

这个问题是本轮继续排查时发现的真实 bug。原因很直接：

- 构建产物 `dist/` 里没有 `config.yaml`
- 但上传模块之前还在浏览器运行时请求 `/config.yaml`

这样一来，开发环境可能正常，生产构建后的上传功能会直接失效。现在已经把这个读取链路改成和其他 service 一致，不再依赖构建产物里额外存在配置文件。

## 7. 修复排序权重字段的数值输入方式

- 修改了 `src/pages/Commands/Commands.tsx`
- 修改了 `src/pages/Practices/Practices.tsx`

具体修复点：

- 把 `Input type="number"` 改成了 `InputNumber`

这样做的原因是 Ant Design 的普通 `Input` 即便写了 `type="number"`，表单值本质上还是字符串。原来的写法有概率把 `sort_order` 按字符串传给后端，导致接口校验失败，或者出现排序权重类型不一致的问题。

# 异常处理手法

- 统一把管理后台请求收口到 `config.ts`，避免每个 service 各自拼 header 时漏掉鉴权
- 文件树重命名增加前置校验，先拦非法输入和重名路径，再执行状态更新
- 文档解析进度回调对空值做兼容，避免前端状态流里出现不符合预期的类型
- 上传配置直接复用内存中的已解析配置，避免生产构建依赖一个不存在的静态配置文件
- 数值字段改为 `InputNumber`，减少表单层把数字误传成字符串的风险

# 可能影响范围

- 管理后台所有通过 service 层发起的接口请求
- 系统技能页面里的附件树编辑
- 文档上传后的解析状态展示
- 指令管理和最佳实践管理里的排序权重录入
- 附件上传到 OSS 的整条链路

这次改动是围绕现有结构做的最小修复，没有改页面布局，也没有改后端接口路径。

- 这个问题已经超出“最小范围修 bug”的范畴，更像架构和安全设计问题
- 更稳妥的方向是把敏感 token 下沉到后端代理层，前端只调受控后端接口，不直接持有长期凭证

# 改动背景（智能体头像链路修复）

这轮是继续处理智能体头像相关的真实问题。用户反馈的现象有两层：

- 智能体列表和详情里看不到头像
- 上传或生成完头像后，智能体接口返回的 `avatar_url` 还是 `null`

这次没有直接拍脑袋改展示，而是先按接口和前端提交流做了排查。排查结论很明确：

- 后端 OpenAPI 里 `POST /api/v1/admin/agents` 和 `PUT /api/v1/admin/agents/{agent_id}` 都明确支持 `avatar_url`
- 当前页面里列表没有头像列，详情也只是把 `avatar_url` 当纯文本显示
- 更关键的是，头像选择器虽然会通过 `onChange` 回填值，但为了避免 antd 自定义组件和表单联动在弹窗场景里出现丢值，这次额外补了一层显式表单同步和提交兜底

# 主要修改点（智能体头像链路修复）

## 1. 补上智能体列表和详情里的头像展示

- 修改了 `src/pages/Agents/Agents.tsx`

具体改动：

- 列表新增了“头像”列
- 统一抽了 `AgentAvatar` 预览组件，列表、表单预览、详情弹窗都复用这一套显示
- 详情弹窗里的“头像”从原来的纯文本改成了图片预览 + 地址展示
- 详情里额外保留了可复制的头像地址，方便排查线上数据

这样就算头像地址存在，页面也会真正把图片渲染出来，不会再只显示一串 URL。

## 2. 修复头像值在创建和编辑时可能丢失的问题

- 修改了 `src/pages/Agents/Agents.tsx`

具体改动：

- 在头像选择器里新增了 `syncAvatarValue()`，选图、本地上传、AI 生成、清空头像时都会显式执行 `form.setFieldValue('avatar_url', ...)`
- 创建和编辑提交时，统一走 `buildAgentSubmitPayload(...)`
- 这个方法会再次从表单里读取 `avatar_url`，并做一次空字符串转 `null` 的整理

这里的处理思路很简单：

- 第一层，在用户选完头像时，强制把值写回表单
- 第二层，在真正提交给后端前，再从表单里拿一次最终值

这样可以最大限度避免头像值只停留在组件内部，最后没有真正进请求体。

## 3. 补回归测试，锁住头像链路

- 修改了 `tests/regressions.test.mjs`

新增回归点：

- 智能体列表里必须存在头像列
- 智能体详情里必须存在头像预览/地址展示
- 创建和编辑提交时，代码里必须显式处理 `avatar_url`

这次先让回归红灯，再补实现，最后重新跑绿。

# 异常处理手法（智能体头像链路修复）

- 先用真实 OpenAPI 确认后端字段名，排除“前端字段传错”的可能
- 再查真实智能体列表接口，确认当前线上确实存在 `avatar_url: null`
- 前端层面通过“组件同步 + 提交兜底”两层保障，降低表单值丢失的风险
- 详情里保留头像地址文本和复制能力，方便继续核对后端实际返回

# 可能影响范围（智能体头像链路修复）

- 智能体编辑弹窗

- 智能体详情弹窗

这次没有改 agent service 的接口定义，也没有改后端字段，只是在前端把头像值同步和展示补完整。

# 追加修复（智能体详情接口返回层级）

这轮继续处理智能体详情弹窗里的一个真实问题。用户已经给出明确证据：

也就是说，这次不是图片组件本身没渲染，而是详情接口的数据层级和列表接口不一样，前端之前直接把 `data` 当成 `AdminAgent` 用了，所以详情弹窗里整块基础信息都取空了。

- 修改了 `src/services/agentService.ts`

- 在 `agentService` 里新增 `normalizeAgentDetailData(...)`
- `fetchAgentDetail(...)` 不再直接把 `raw.data` 原样返回
- 现在会先兼容判断：
  - 如果是 `data.agent`，就取 `data.agent`
  - 如果后端后面改成平铺结构，也还能继续兼容 `data`

这样页面层就不用再关心详情接口到底包不包一层 `agent`，列表和详情都能继续共用同一套 `AdminAgent` 类型。

# 改动背景（图片管理）

这轮新增了后台里的“图片管理”页面，用来统一维护系统里的图片资源。范围按当前需求收口，只接入图片列表、详情、更新、删除、上传 5 个接口，不接入 `/api/v1/admin/images/generate`。

图片管理页这次优先覆盖两类真实业务用途：

- 技能图片
- 智能体图片

考虑到后端当前只有 `category` 和 `tags` 两层信息，没有单独的“业务归属”字段，所以前端这次采用“主类目 + 用途标签”的方式做分类：

- `category` 负责图片主类目：`avatar / icon / banner / misc`
- `tags` 里保留 `skill` 或 `agent`，用于区分技能图片和智能体图片

# 主要修改点（图片管理）

## 1. 新增图片管理路由和左侧菜单

## 2. 新增图片管理 service，统一接入图片接口

接入接口如下：

- `GET /api/v1/admin/images`

- 详情弹窗：查看图片元数据和访问地址

页面统一使用了 antd 组件，没有引入额外 UI 依赖。

# 追加改动（图片管理接入 AI 生图）

这轮是在已有图片管理页基础上，补接了 `/api/v1/admin/images/generate` 接口。

本次仍然保持最小范围改动，没有重做图片管理页结构，只是在原来的“刷新 / 上传图片”操作区里，新增了一个“AI 生图”入口。生成成功后直接刷新图片列表，方便用户马上在图片库里看到新资源。

## 主要修改点（AI 生图）

### 1. 图片 service 补接 AI 生图接口

- 修改了 `src/services/imageService.ts`

新增了：

- `GenerateImagePayload`
- `generateImage(...)`

请求体按后端当前 OpenAPI 字段接入：

- `prompt`
- `style`
- `size`
- `name`
- `category`
- `tags`

其中：

# 追加改动（智能体头像改为图片库三路径）

这轮把智能体创建/编辑弹窗里的头像输入，从原来的纯文本 `头像URL` 改成了统一的头像选择器。目标是把头像来源统一收口到图片管理体系里，避免用户手动填 URL。

现在头像支持 3 条路径：

- 从图片管理库选择
- 本地上传后入库，再回填
- 根据智能体名称、描述、提示词拼接内容走 AI 生图，生成后入库，再回填

这里对后端接口保持最小改动：

# 追加修复（智能体头像上传成功但未回填）

这轮修了智能体头像选择器里的一个真实 bug：图片上传接口其实已经返回了图片地址，但前端没有从正确位置取值，所以页面误提示“图片已上传到图片库，但返回结果里没有拿到图片地址”。

## 根因

实际接口返回结构里，图片地址在：

```json
data.image.url
```

而之前前端只在这些平级字段里取值：

- `data.url`
- `data.image_url`
- `data.file_url`
- `data.oss_url`

所以虽然上传成功、图片也确实落库了，但头像地址没有成功回填到智能体表单。

## 本次修复点

- 修改了 `src/pages/Agents/Agents.tsx`

具体处理：

- 新增了统一的 `extractImageUrl(...)`
- 上传头像时兼容从 `data.image.url` 取地址
- AI 生图后回填头像时也统一走这套地址提取逻辑

这样上传和 AI 生成两条路径都不会再漏掉嵌套返回结构。

## 顺手收敛的交互

用户反馈头像区域顶部那层模式切换没实际价值，这轮一起去掉了。

保留后的可用入口还是这 3 个：

- 从图片库选择
- 本地上传
- AI 生成

但不再额外显示一层和按钮功能重复的切换条。
